# Claude Rules System

## Where rules and commands live

`.claude-static/` in the `.github` repo is the source of truth for all org-wide rules and commands. `.claude/` is the local working copy, derived from `.claude-static/`. Distribution syncs `.claude-static/` to all NXD repos.

**Locality is determined by location, not by filename suffix.** A file present in `.claude-static/` is distributed. A file present only in `.claude/` is local to that repo. No special suffix is required to opt out of distribution — simply do not add the file to `.claude-static/`.

## Workflow

All changes to org-wide rules and commands start in `.claude-static/`. After editing, derive the local copy:

```bash
cp .claude-static/<subdir>/<file>.md .claude/<subdir>/<file>.md
```

Both files ship in the same commit and PR. The `validate-static-parity` CI workflow enforces this — a PR that changes `.claude-static/` without a matching `.claude/` update will fail.

## Local testing

`~/.claude/` may be used for local testing before submitting a PR. The rules owner decides what enters `.claude-static/` — Claude does not self-promote.

## Triggers

Before committing any new or changed file under `.claude-static/`, verify the local copy has been derived to the corresponding `.claude/` path.

For `.gen.md` files: autogenerated — never edit manually.

For rule files (`.md` under `.claude-static/rules/`, excluding `.gen.md` and `README.md`): also check whether `.claude-test/rules/<subfolder>/<filename>/` directory exists. If the directory exists and the test file does not, flag it as a coverage gap before committing. In repos without `.claude-rule-tests/`, this check does not apply.

## Proposing new rules

When Claude identifies a missing or needed rule, it notifies the local user: "I have identified a Claude rules gap: `<finding>`. Shall we request a change?" On approval, Claude opens a PR to `.github`. The PR description must cover:
- **Goal** — one sentence: what problem does this solve or what outcome does it achieve?
- **What** the rule is
- **Why** it is needed
- **Value** it adds
- **Risks** any concerns or potential side effects

## Deriving local copy from .claude-static/

Changes start in `.claude-static/`. Derive the local copy in the same branch — never as a separate PR.

```bash
cp .claude-static/<subdir>/<file>.md .claude/<subdir>/<file>.md
```

Stage alongside the `.claude-static/` change and commit together.

Every `.md` file in `.claude-static/` must have a corresponding copy in `.claude/`. Exception:
- `.gen.md` — autogenerated, derive via generation script, not manually

Files that exist only in `.claude/` (not in `.claude-static/`) are local to the repo by virtue of their location — no special suffix required.

## Blast radius

Changes to `.claude-static/` are distributed to all repos in the NXD-Solutions org. Review with extra care before merging.

PR description must cover **Goal**, **What**, **Why**, **Value**, **Risks**. Blast radius assessment must identify which repos are affected and whether any may not match the rule's assumptions.

## Commands

Command files live in `.claude/commands/`. When writing the `description` frontmatter field, end the description with `(NXD)` — this makes the command discoverable when typing `/nxd` in the Claude Code command picker.

Org-wide commands follow the same distribution pattern as rules: the trigger fires, and approved commands are promoted to `.claude-static/commands/` in `.github`. Repo-specific commands live only in `.claude/commands/` and are never added to `.claude-static/`.

### Command structure

Every command is a Steward. H1 headings are reserved for structural divisions — never use H1 for anything else.

| H1 | Purpose | Count |
|---|---|---|
| `# Identity` | Duties, services, session preamble — who this Steward is | Exactly one |
| `# Foundation` | Shared knowledge, constraints, and invariants across all duties | Exactly one |
| `# <Duty>` | One section per duty — named after the duty it serves | One per duty |

## Autogenerated files

Files with the `.gen.md` suffix are autogenerated — never edit them manually.
